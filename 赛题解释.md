# 赛题解释（天池：资金流入流出预测）

## 1. 这个赛题本质上要做什么

这个题的核心是：**根据历史用户行为数据，预测未来每天的总申购量和总赎回量**。  
其中：
- 申购：用户把银行卡等渠道的钱转入余额理财账户（可理解为“买入理财份额”）。
- 赎回：用户把余额理财账户里的钱转出到账户余额或银行卡（可理解为“卖出理财份额并取回现金”）。
从你目录中的 `comp_predict_table.csv` 可以看出，最终要提交的是按天的两列预测值（示例里是 2014-09-01 到 2014-09-03）。

你可以把它理解为一个**按日期的时间序列预测问题**，但原始数据是用户级交易明细，所以通常要先做“用户明细 -> 天级聚合特征”。

---

## 2. 你当前数据文件分别是什么

你的 `Purchase Redemption Data` 目录里有 5 个文件（常见是 4 个数据表 + 1 个提交样例）：

1. `user_balance_table.csv`（主表，最重要）  
   - 粒度：用户-日期  
   - 关键字段：
     - `user_id`, `report_date`
     - `total_purchase_amt`：当日总申购
     - `total_redeem_amt`：当日总赎回
     - 以及 `consume_amt`、`transfer_amt`、余额字段等
   - 用途：训练标签 + 绝大多数行为特征都来自这里

2. `user_profile_table.csv`（用户画像表）  
   - 粒度：用户  
   - 字段：`sex`, `city`, `constellation`
   - 用途：做人群分层特征（例如不同城市、性别的交易习惯）

3. `mfd_day_share_interest.csv`（收益率序列）  
   - 粒度：日期  
   - 字段：`mfd_daily_yield`, `mfd_7daily_yield`
   - 用途：市场收益环境特征，辅助解释申购/赎回变化

4. `mfd_bank_shibor.csv`（Shibor 利率序列）  
   - 说明：Shibor（上海银行间同业拆放利率）是银行之间短期借贷资金的市场利率，可视作资金价格的“宏观温度计”。
   - 粒度：日期  
   - 字段：隔夜、1周、1月、3月等期限利率
   - 用途：宏观利率环境特征

5. `comp_predict_table.csv`（提交格式示例）  
   - 内容类似：`report_date,purchase,redeem`
   - 用途：告诉你**最终输出文件格式**

---

## 3. 任务输入与输出（你实际要交什么）

- **输入**：上述历史数据（用户行为 + 用户画像 + 收益率 + 利率）
- **输出**：测试时间段内每天两列预测值  
  - `purchase`：当日总申购预测
  - `redeem`：当日总赎回预测
- **提交格式**：与 `comp_predict_table.csv` 一致（按天一行）

---

## 4. 你想要的技术路线：Python + MySQL（SQL 为主处理）

你这个思路非常适合本题，建议职责分工如下：

- **MySQL/SQL 负责**：
  - 数据入库与清洗
  - 天级聚合
  - 构造滞后特征（lag）、滑窗均值、周几/月初月末等日历特征
  - 多表 join（收益率、Shibor）
- **Python 负责**：
  - 读取 SQL 结果集
  - 训练模型（如 LightGBM / XGBoost / CatBoost / ARIMA 类）
  - 验证与调参
  - 生成提交文件

## 5. 推荐的实战流程（按这个顺序做）

## 第一步：建表并导入 CSV

建议建 5 张表，字段类型注意：
- 日期字段用 `DATE`（CSV 里是 `YYYYMMDD`，入库时转成日期）
- 金额字段用 `DECIMAL(20,2)` 或 `DOUBLE`
- 类别字段（如星座）用 `VARCHAR`

---

## 第二步：先做“按天标签表”

从 `user_balance_table` 聚合出每天全市场总量（这是监督学习标签；监督学习标签=模型要学习去预测的目标值）：
- `y_purchase = sum(total_purchase_amt)`
- `y_redeem = sum(total_redeem_amt)`

这一步之后你会得到核心序列：`date, y_purchase, y_redeem`。

---

## 第三步：做时间序列特征（SQL 可完成大部分）

重点特征建议：

1. **滞后特征**
   - `lag1/lag2/lag3/lag7/lag14/lag30`（对 purchase 和 redeem 都做）

2. **滑窗统计**
   - 说明：滑窗（rolling window）就是“只看最近 N 天”的局部统计，例如近 7 天均值。
   - 3/7/14/30 天均值、标准差、最大最小值

3. **日历特征**
   - `dayofweek`、是否周末、是否月初/月末、月份等

4. **外部市场特征**
   - 关联 `mfd_day_share_interest`
   - 关联 `mfd_bank_shibor`

5. **可选人群特征**
   - 用户画像 join 后，按天聚合不同分组的申赎金额占比

---

## 第四步：划分训练/验证集

不要随机切分，要按时间切分，例如：
- 训练：较早时间段
- 验证：后面连续一段时间（模拟线上预测未来）

这样评估才贴近比赛场景。

---

## 第五步：建模与预测

可先做两个独立模型：
- 模型 A 预测 `purchase`
- 模型 B 预测 `redeem`

常见做法是树模型（对复杂非线性和多特征较稳）。  
若你想进一步提升，可尝试：
- 分别建工作日/周末模型
- 预测结果做平滑或规则修正（节假日附近）

---

## 第六步：导出提交文件

最终导出 3 列：
- `report_date`（如 `20140901` 格式）
- `purchase`
- `redeem`

并确保与 `comp_predict_table.csv` 的列顺序一致。

---

## 6. 你可以先明确的“最低可跑通版本”

如果你想先快速打通全流程（MVP，Minimum Viable Product，最小可行版本），可以用这套最小方案：

1. 只用 `user_balance_table` 做天级聚合  
2. 做 `lag1/lag7/lag14` + `dayofweek` 特征  
3. 用 Python 训练两个回归模型  
4. 输出提交文件

先跑通，再逐步加上收益率、Shibor、用户画像特征，效果通常会明显提升。
